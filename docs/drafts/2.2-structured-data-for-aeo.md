## 2. Контекст / Проблема

- «Додай Schema — потрапиш в AI Overviews» — поширений, але неточний меседж.
- Без розмітки сутності передані пошуковій системі менш явно — але це не умова AI-видимості.
- Частина схем (FAQ rich results) більше не дає Google rich-result ефекту.

## 3. Ядро

### 3.1. Що структура реально дає

Розмітка не «проштовхує» сторінку — вона дає explicit clues. Документація Google зі structured data пояснює, що розмітка допомагає системі розуміти зміст сторінки й може дати eligibility для rich results [Джерело 68]. Тобто це технічна ясність і Search-eligibility, а не важіль, що гарантує цитування в AI-відповіді.

### 3.2. Для AI Overviews / AI Mode спец-схеми немає

Тут найчастіший overclaim. Google Search Central прямо документує, що для AI Overviews / AI Mode немає спеціальної schema.org-розмітки й окремих технічних вимог понад базове SEO [Джерело 65]. Висновок: не продавайте «AI-схему» — її не існує; працює та сама технічна база, що й для звичайного Search.

### 3.3. FAQPage більше не Google rich-result важіль

Контент-блок FAQ корисний, але JSON-LD `FAQPage` змінив статус. За документацією Google, FAQ rich results більше не зʼявляються в Google Search із 07.05.2026, а звіти й підтримку прибирають [Джерело 68]. Практика: видимий FAQ-контент лишаємо, `FAQPage` — лише опціональна семантика (і для не-Google споживачів), а не Google rich-result lever.

### 3.4. Машинна читабельність — окрема мета

Сайт читають не лише люди. Salemi et al. розглядають пошук не тільки для людей, а й для машинних споживачів, включно з RAG-системами [Джерело 17]. Для GEO це означає: чіткі сутності, визначення й структуровані факти роблять контент придатним для машинного використання — незалежно від статусу конкретного rich-result.

### 3.5. Публічний сайт як джерело для асистентів

Структурований публічний контент живить корпоративні асистенти. Microsoft Copilot Studio документує, як Copilot може використовувати публічні сайти для generative answers [Джерело 73]. Це показує практичний шлях: ясно структурований публічний контент стає джерелом не лише для пошуку, а й для кастомних AI-асистентів.

## 5. AEO / FAQ

### Чи допоможе structured data потрапити у відповідь AI?
Розмітка дає explicit clues і Search-eligibility (rich results), але це не прямий AI-буст: спеціальної schema під AI Overviews / AI Mode немає, як і вимог понад базове SEO [Джерело 65, 68].

### Чи варто додавати JSON-LD FAQPage?
Видимий FAQ-контент — так. JSON-LD `FAQPage` — лише опціонально: FAQ rich results прибрані з Google Search із 07.05.2026, тож це вже не Google rich-result важіль [Джерело 68].

### Чи є окрема «schema під AI»?
Ні. Google прямо документує: для AI Overviews / AI Mode немає спеціальної schema.org-розмітки й окремих технічних вимог понад базове SEO [Джерело 65].

### Навіщо тоді structured data для GEO?
Вона робить сутності й факти явними для машинного використання, зокрема RAG-систем і кастомних асистентів — це ясність, а не гарантія цитування [Джерело 17, 73].

## 6. Schema (плейсхолдери)

> Псевдокод-плейсхолдери для ревʼю. У production — валідний JSON-LD без коментарів. **Увага:** FAQ rich results вилучені з Google Search із 07.05.2026 — `FAQPage` лише як опціональна семантична розмітка (не Google rich-result lever); основна схема — `Article`. Видимий FAQ-контент зберігаємо. **У коді:** `Article` завжди; видимий FAQ завжди; `FAQPage` — опціонально / за feature-flag; коментарі `//` та `/* */` видалити, `mainEntity` заповнити.

```jsonld
// Article
{ "@context":"https://schema.org", "@type":"Article",
  "headline":"Structured data для AEO: що розмітка дає, а що ні",
  "about":["structured data","schema.org","AEO"],
  "dateModified":"{{UPDATED_AT}}" }
```
