## 1. Hero (H1)

**H1:** Технічне SEO як фундамент GEO

**Підзаголовок:** GEO не скасовує технічного SEO — воно стоїть на ньому. Якщо сайт недоступний або погано структурований, оптимізація під AI не спрацює.

## 2. Контекст / Проблема

- Недоступний для crawling контент важко вилучити; robots.txt контролює доступ краулера, але не видаляє URL із Search.
- Без structured data сутності менш явно передані пошуковій системі (це допомога розумінню, не умова AI-видимості).
- GEO без технічної бази — це оптимізація того, чого пошукова система не бачить.

## 3. Ядро

### 3.1. Базовий рівень не змінився

Crawling, indexing і ranking лишаються першим шаром видимості. Brin & Page описали саме цю архітектуру як основу пошуку [Джерело 57], а офіційний стартовий гайд Google підтверджує: crawlability, indexability, якість і структура — базові умови [Джерело 67]. GEO додає шар генерації поверх цього рівня, а не замість нього.

### 3.2. Офіційна позиція платформи

Орієнтуватися треба на першоджерела, а не на інтерпретації агентств. Google Search Central прямо документує, як AI-функції в Search взаємодіють із веб-контентом [Джерело 65]. Це первинний документ для розуміння AI Overviews і AI Mode у Google-екосистемі — базовий орієнтир для технічного GEO-аудиту.

### 3.3. Доступність для краулерів

Якщо краулер не може отримати доступ до сторінки, її важче вилучити. Огляд Google crawlers and fetchers пояснює, які агенти Google використовують для різних задач [Джерело 76]. robots.txt керує доступом краулера, але не є механізмом утримання сторінки поза Google: заблокований URL усе одно може зʼявитися без опису, якщо на нього посилаються інші сторінки. Для повного виключення з Search потрібні `noindex` або password protection. Перевірка доступу — одна з перших і найдешевших у GEO-аудиті.

| Перевірка | Ризик при провалі | Джерело |
|-----------|--------------------|---------|
| robots.txt і доступ агентів | Контент недоступний для crawling/extraction; URL може зʼявлятися без опису. Повне виключення — `noindex`/пароль | 76 |
| Crawlability / indexability | Сторінки немає у видачі й відповіді | 67 |
| Structured data | Слабке машинне розуміння сутностей | 68 |
| Відповідність AI-документації Google | Розрив із реальною поведінкою Search | 65 |

### 3.4. Structured data — не магія, а ясність і Search-eligibility

Розмітка не «проштовхує» сторінку: вона дає explicit clues і допомагає Google зрозуміти сторінку, може дати eligibility для rich results [Джерело 68]. Без неї сутності менш явно передані, але це не обовʼязкова умова AI-видимості. Важливо: для AI Overviews / AI Mode немає спеціальної schema.org-розмітки й окремих технічних вимог понад базове SEO [Джерело 65]. Тобто structured data — ясність і Search-eligibility, а не прямий AI-буст. Окремо: FAQ rich results вилучені з Google Search із 07.05.2026 — `FAQPage` лишаємо лише як опціональну семантику.

### 3.5. SEO-патерни розпізнаються — для GEO це ймовірний ризик

Маніпулятивні патерни виявляються. Lewandowski & Yagci пропонують багатовимірний підхід до виявлення SEO-патернів у результатах пошуку [Джерело 60]. Аналогія для GEO — обережна гіпотеза, а не доведений факт: штучні патерни оптимізації, ймовірно, з часом класифікуватимуться платформами, тож технічна база й якість надійніші за обхідні трюки.

## 5. AEO / FAQ

### Чи замінює GEO технічне SEO?
Ні. Crawlability, indexability, structured data й доступність для краулерів лишаються базовим рівнем; GEO додає шар генерації поверх нього, а не замість [Джерело 67].

### Що перевіряти першим у GEO-аудиті?
Доступ краулерів (robots.txt, firewall): неправильне налаштування ускладнює crawling/extraction. Але robots.txt — це контроль доступу, а не видалення з Search; для виключення сторінки потрібні `noindex`/пароль [Джерело 76].

### Чи допоможе structured data потрапити у відповідь AI?
Розмітка дає explicit clues і Search-eligibility (rich results), але це не прямий AI-буст: спеціальної schema під AI Overviews / AI Mode немає, як і вимог понад базове SEO. FAQ rich results прибрані з 07.05.2026 — `FAQPage` лише опціональна семантика [Джерело 65, 68].

### На чию документацію спиратися?
На першоджерела платформ: Google Search Central прямо документує взаємодію AI-функцій Search із веб-контентом — це надійніше за інтерпретації агентств [Джерело 65].

## 6. Schema (плейсхолдери)

> Псевдокод-плейсхолдери для ревʼю. У production — валідний JSON-LD без коментарів. **Увага:** FAQ rich results вилучені з Google Search із 07.05.2026 — `FAQPage` лише як опціональна семантична розмітка (не Google rich-result lever); основна схема — `Article`. Видимий FAQ-контент зберігаємо. **У коді:** `Article` завжди; видимий FAQ завжди; `FAQPage` — опціонально / за feature-flag; коментарі `//` та `/* */` видалити, `mainEntity` заповнити.

```jsonld
// Article
{ "@context":"https://schema.org", "@type":"Article",
  "headline":"Технічне SEO як фундамент GEO",
  "isPartOf":{"@type":"Book","name":"GEO Playbook"},
  "about":["technical SEO","crawlability","structured data"],
  "citation":[
    {"@type":"CreativeWork","name":"AI features and your website","url":"https://developers.google.com/search/docs/appearance/ai-features"},
    {"@type":"CreativeWork","name":"Introduction to structured data markup in Google Search","url":"https://developers.google.com/search/docs/appearance/structured-data/intro-structured-data"}
  ],
  "dateModified":"{{UPDATED_AT}}" }
```

// FAQPage — мапиться з блоку 5 (4 Q&A)
